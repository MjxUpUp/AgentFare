## 一、AgentOpt — 唯一真正做 Per-step 的工具

### 1.1 基本事实

| 指标 | 数据 |
|------|------|
| 包名 | `pip install agentopt-py`（PyPI）/ `pip install trace-opt`（内部包名） |
| License | Apache 2.0 |
| GitHub | github.com/AgentOptimizer/agentopt |
| 来源 | Columbia University DAPLab（注意：原报告称"微软研究院+哥伦比亚+康奈尔"，但 GitHub 仅标注 Columbia） |
| 文档 | agentoptimizer.github.io/agentopt |

### 1.2 技术架构

**核心机制**：在 httpx 传输层拦截 LLM 调用——所有 LLM SDK（OpenAI、Anthropic、Gemini）底层都走 httpx，AgentOpt 在这个唯一咽喉点做拦截。

```
your_agent(input)
  └── framework internals (LangChain, CrewAI, etc.)
        └── httpx.Client.send()   ← 在这里拦截
              └── LLM API (OpenAI, Anthropic, etc.)
```

**用户需要提供 4 样东西**：
1. **Agent 类**：实现 `__init__(self, models)` 和 `run(self, input_data)`
2. **数据集**：`(input_data, expected_output)` 对的列表
3. **评估函数**：`eval_fn(expected, actual) → score`
4. **候选模型**：`{"step_name": ["model_a", "model_b", ...]}` 字典

### 1.3 搜索算法

| 算法 | 适用场景 | 原理 |
|------|---------|------|
| `brute_force` | 小搜索空间 | 遍历所有组合 |
| `random` | 快速探索 | 随机采样 |
| `hill_climbing` | 拓扑感知搜索 | 基于模型质量/速度排名的贪心搜索 |
| `arm_elimination` (默认 auto) | 最佳臂识别 | Bandit 算法，消除统计上劣势的组合 |
| `epsilon_lucb` | 节省评估预算 | 找到 ε-最优即停止 |
| `threshold` | 阈值判断 | 判断每个组合是否超过用户定义的性能阈值 |
| `lm_proposal` | LLM 引导搜索 | 用 proposer LLM 短列候选组合 |
| `bayesian` | 昂贵评估场景 | 基于高斯过程的贝叶斯优化 |

### 1.4 实际节省数据（README 声明）

| Benchmark | 昂贵组合 | 精度 | 成本 | 省钱组合 | 精度 | 成本 | 节省 |
|-----------|---------|------|------|---------|------|------|------|
| BFCL | Opus | 72% | $60.78 | Qwen3 Next | 71% | $1.87 | 32x |
| HotpotQA | Opus+Opus | ~73% | $2.71 | Qwen3 Next+gpt-oss-120b | 71.3% | $0.13 | 21x |
| MathQA | Opus+Opus | ~98.5% | $5.89 | Ministral+C3 Haiku | 94.0% | $0.05 | 118x |

### 1.5 关键局限性

1. **纯离线工具**：必须先准备 eval 数据集，跑完搜索后才能得到最优组合，没有实时路由能力
2. **无生产部署模式**：没有 web server、没有 API endpoint、没有 streaming
3. **用户需手动定义步骤**：不能自动检测 agent pipeline 中的步骤边界
4. **搜索空间有限**：3步×8模型=512组合，更多步骤/模型时需要更智能的算法
5. **无在线学习**：一旦确定最优组合就固定，不会从生产流量中持续优化
6. **维护活跃度存疑**：GitHub 上commits频率需要持续关注

### 1.6 竞争性判断

AgentOpt 是一个**优秀的学术验证工具**，但离商业产品差距很大。它的价值在于：
- 验证了 per-step routing 的可行性和巨大节省空间
- 提供了可复用的搜索算法
- Apache 2.0 许可证意味着我们可以借鉴其算法思路

**但它不是我们的竞争对手——它是我们项目的技术基础之一。**

## 二、LiteLLM — AI 界的 Nginx

### 2.1 路由能力验证

| 路由策略 | 实际能力 | 与 Per-step 的关系 |
|---------|---------|-------------------|
| `simple-shuffle`（默认） | 随机分发到配置的多个模型部署 | 无关 |
| `least-busy` | 发到最空闲的部署 | 无关 |
| 成本感知路由 | 优先选择成本低的 | Per-query 级别 |
| 延迟感知路由 | 优先选择延迟低的 | Per-query 级别 |
| 健康检查驱动 | 自动移除不健康部署 | Per-query 级别 |
| 自适应路由 (BETA) | 根据实时指标调整 | Per-query 级别 |
| 标签路由 | 按请求 tag 分配 | Per-query 级别 |
| Fallback | 失败后切到备选模型组 | Per-query 级别 |
| 预算路由 | 按 team/key 设预算上限 | 计费层面 |

### 2.2 关键发现

- **100% 是 Per-query 级别**：没有任何 pipeline 感知、步骤感知或组合优化能力
- **"Agent Platform"** 在导航栏出现，但实际上只是 agent 可以通过 LiteLLM proxy 调用多模型，不是 agent-aware routing
- **$10M+ ARR** 证明了 LLM Gateway 市场的真实需求
- **社区驱动**：约 8 人团队，纯开源社区增长到 $10M ARR

### 2.3 竞争性判断

LiteLLM 不是我们的直接竞争对手——它是**互补关系**。
- 我们的 per-step routing 引擎应该运行在 LiteLLM（或类似 gateway）之上
- LiteLLM 解决"统一入口"问题，我们解决"pipeline 级最优组合"问题
- **可能合作而非竞争**：LiteLLM 可能愿意集成我们的 per-step routing 作为高级功能

## 三、OpenRouter — 通用 LLM 适配器

### 3.1 路由机制

| 后缀 | 作用 | 原理 |
|------|------|------|
| `:nitro` | 最高吞吐量 | 按 provider throughput 排序选最快的 |
| `:floor` | 最低价格 | 自动选最便宜的 provider |
| `:online` | 联网搜索 | 选择有 web 搜索能力的 provider |

**本质**：OpenRouter 的路由是 **provider 级别的负载均衡**，不是模型级别的智能选择。同一个模型可能有多个 provider（如 OpenAI 官方 vs Azure），OpenRouter 选最优 provider。

### 3.2 关键数据

- ARR 从 $5M（2025.5）→ $50M（2026年初），10 个月 10x 增长
- 处理了 100T+ token 的元数据——**这是最大的 LLM 使用模式数据库**
- 融资 $40.5M（a16z、Menlo Ventures、Sequoia）
- 正在洽谈 $120M 融资 @ $1.3B 估值

### 3.3 竞争性判断

OpenRouter 是 API 聚合层，不涉及 pipeline 优化。
- **数据是壁垒**：100T+ token 的使用数据有巨大价值，可能未来用于智能路由
- **目前不做 per-step**：但以他们的数据规模，如果要做会很快
- **我们的定位**：可以在 OpenRouter 之上运行，利用其多模型接入能力

## 四、Not Diamond — 聪明的 Per-query 路由

### 4.1 实际能力

- **核心产品**：Per-query 智能路由，根据请求难度/质量/成本/延迟自动选择最优模型
- **自定义路由训练**：提供框架，让用户在自己的 eval 数据上训练路由算法
- **"Agent Routing" 宣称**：官网有 "Model Routing for Agents" 页面，但实际是每个 agent 的请求独立路由，不是 pipeline 级组合优化
- **40+ providers** 支持
- **融资**：$2.3M Seed

### 4.2 竞争性判断

Not Diamond 最接近我们的方向但仍有根本差距：
- 他们的路由粒度是"每个独立的 LLM 调用"
- 我们的路由粒度是"整个 agent pipeline 的步骤组合"
- **他们不知道步骤间的耦合依赖**——planner 的选择影响 solver 的表现，这种组合效应 Not Diamond 完全不考虑

## 五、Martian — 动态路由 + 可解释性

### 5.1 实际能力

- **核心专利**："Model Mapping"——将不可解释的模型内部矩阵转化为可理解的跨模型映射
- **Accenture 集成**：已集成到 Accenture 的 "switchboard" 服务
- **Airlock 合规**：推出了企业合规工具
- **融资**：$9M Seed（NEA、General Catalyst、Prosus Ventures）
- **32 人团队**

### 5.2 竞争性判断

Martian 的优势在可解释性和企业合规，不在 per-step 优化。
- Accenture 投资是一个强信号——大型咨询公司需要帮客户优化 LLM 成本
- **可解释性是我们的差异化机会**：如果我们的 per-step routing 也能提供可解释的路由决策（为什么这步用 Haiku 不用 Opus），会更有竞争力

## 六、SkyAPI（UIUC 学术） — Stage-level 编排

### 6.1 技术概要

| 指标 | 数据 |
|------|------|
| 来源 | UIUC Siebel School，Phillip Wang 硕士论文 |
| 发表时间 | 2025-12 |
| 核心方法 | MILP（混合整数线性规划）做 stage-level 编排 |
| 关键创新 | Prefix-aware 调度 + TTFT 延迟，最大化 KV-cache 复用 |
| 验证基准 | DeepResearch、Gama-Bench |
| 效果 | 成本降低 3x，满足严格延迟约束 |

### 6.2 与我们方向的关系

SkyAPI 的"stage-level"和我们的"per-step"概念最接近：
- SkyAPI 将 agent workflow 分解为有序执行阶段（stage），在每个阶段级别优化 provider 选择
- 使用 MILP 显式优化 stage makespan 和成本的权衡
- **KV-cache 复用**是一个我们没有考虑到的重要优化维度——协作 agent 间如果共享前缀，可以利用 KV-cache

**局限**：硕士论文级别，没有产品化，没有开源代码。

## 七、中国市场

### 7.1 国产模型 API 定价（2026年5月最新）

| 模型 | 输入 ($/MTok) | 输出 ($/MTok) | 缓存命中 ($/MTok) | 上下文 | 擅长 |
|------|-------------|-------------|-----------------|--------|------|
| DeepSeek V4-Pro | 0.435 | 0.87 | 0.003625 | 128K | 最便宜 token、编程 |
| DeepSeek V4-Flash | 0.14 | 0.28 | 0.02 | 1M | 极低成本 |
| 小米 MiMo V2.5 Pro | 1.00 | 3.00 | 0.20 | 1M | 长文档（唯一1M上下文） |
| 阿里 Qwen3 Max | 0.78 | 3.90 | 0.156 | 262K | 生产环境平衡 |
| 月之暗面 Kimi K2.6 | 0.16-2.00（分层） | ~2.50 | **0.07** | 128K | 编程 agent、缓存友好 |
| 智谱 GLM-5 | 1.00 | 3.20 | 未公开 | 200K | 结构化推理 |

**对比海外旗舰**：DeepSeek V4-Pro 输出 $0.87 vs GPT-5.5 ~$30 → **34倍差价**。这为中国市场的 per-step routing 创造了巨大优化空间。

### 7.2 中国的 LLM Gateway 生态

| 项目 | Stars | 特点 |
|------|-------|------|
| **one-api**（songquanpeng） | 极高（最流行） | 统一 OpenAI 兼容接口、模型映射、自动重试 |
| **new-api**（QuantumNous） | 高 | one-api fork，改进了渠道管理 |
| **metapi** | 中等 | 聚合多个 one-api/new-api 实例，智能路由 |
| **llm-gateway-lite** | 新 | OpenResty 基于，轻量高性能 |

**关键判断**：
- one-api 是中国的 LiteLLM——占据了统一 API 网关的位置
- 所有中国 gateway 都是 **per-query 级别的基础路由**，无 per-step 感知
- **中国市场的 per-step routing 是完全空白**
- 中国开发者已经在做手动版 per-step routing（70-85% 流量到 DeepSeek + 复杂问题到贵模型），但没有工具化

### 7.3 中国市场的特殊性

1. **价格差更大**：DeepSeek V4-Flash ($0.28 output) vs GLM-5 ($3.20 output) = **11倍**，比海外模型的价差更大 → per-step routing 的优化空间更大
2. **缓存定价分化**：DeepSeek 缓存命中 $0.004 vs Kimi $0.07，不同模型缓存策略差异大 → 路由时必须考虑缓存效率
3. **上下文窗口差异大**：MiMo 1M vs DeepSeek 128K → 长文档任务必须路由到特定模型
4. **双模型路由已是常见实践**：开发者已经手动在做简单版 per-step routing，但缺少工具支持
5. **one-api 生态垄断**：任何做中国市场 LLM gateway 的项目都需要考虑 one-api 兼容

## 九、核心结论与项目定位修正

### 9.1 验证后的结论

1. **Per-step agent routing 确实是空白市场**——所有验证都证实了原报告的判断
2. **AgentOpt 可以作为算法基础**——Apache 2.0 许可证，8 种搜索算法可直接复用
3. **LiteLLM/one-api 是基础设施层**——我们应该在它们之上运行，而非替代
4. **中国市场的价差更大** → 优化空间更大 → 双市场策略有道理
5. **SkyAPI 的 KV-cache 复用思路值得借鉴**

### 9.2 差异化定位（验证后修正）

> **"Agent Pipeline 优化引擎"** — 在 LLM Gateway 之上，自动优化 Agent pipeline 中每一步的模型选择。
> 
> 不是又一个 Gateway（LiteLLM/one-api 已做得很好），而是 Gateway 之上的**优化层**。

### 9.3 技术差异化要点

| 维度 | 现有方案 | 我们 |
|------|---------|------|
| 路由粒度 | 每次独立 LLM 调用 | Agent pipeline 的步骤组合 |
| 优化目标 | 单次请求的质量/成本/延迟 | Pipeline 端到端的质量/成本/延迟 Pareto 最优 |
| 步骤间关系 | 不考虑 | 显式建模步骤间的耦合依赖 |
| 路由决策 | 静态规则或 per-query 学习 | 离线搜索最优组合 + 在线微调 |
| 中国模型 | 很少考虑 | 一等公民支持 |
| KV-cache 感知 | 无 | 借鉴 SkyAPI，考虑前缀共享 |
